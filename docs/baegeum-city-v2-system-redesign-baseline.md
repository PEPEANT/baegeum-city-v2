# 배금도시 v2 재설계 기준선

결론: 지금부터 배금도시 v2는 기능 목록이 아니라 하나의 검증 가능한 루프로 다시 설계한다. 기준 루프는 `배금시티 생활 준비 -> 버스 이동 -> 다이스시티 도박 -> ledger 정산 -> 폰/인터넷 확인 -> 배금시티 복귀`다.

## 왜 다시 잡는가

현재 프로젝트는 좋은 재료가 많다.

- 멀티맵: `baegeum-city`와 `dice-city`가 분리되어 있다.
- 도시 도구: 맵에디터, 건물 셸, 장식물, draft key 분리가 있다.
- 런타임: 건물 입장, 테이블 착석, 환전 ATM, 홀짝 예약이 있다.
- 경제: `cash / bank / chips / ledger`와 실패 관측성이 있다.
- 시간: `WorldClock`과 반복 이벤트 기준이 있다.
- 폰/인터넷: DIS 미리보기와 MammonCity 폰 셸이 있다.

문제는 재료가 아니라 순서다. 지금 기능을 더 붙이면 “돈은 움직이는데 왜 움직였는지 모르는 게임”이 될 수 있다. 그래서 재설계 기준선은 새 기능보다 검증 가능한 루프를 우선한다.

## 게임의 1차 정체성

배금도시 v2는 GTA식 거대한 도시가 아니라, 처음에는 두 도시의 역할이 분명한 경제/도박 도시 프로토타입이다.

```text
배금시티
생활, 준비, 회복, 현금, 편의시설, 집, 경찰/부동산/주식시장

다이스시티
도박, 밤거리, 위험, 칩, 카지노, 전당포/사채/모텔/유흥
```

이 분리는 단순 맵 분리가 아니라 경제/온라인/채팅/랭킹 분리 기준이다.

## 첫 완성 루프

첫 완성 루프는 아래 8단계다.

1. 플레이어가 배금시티에서 시작한다.
2. 현재 현금, 칩, 배고픔/에너지, 가방 상태를 HUD에서 본다.
3. 버스터미널로 다이스시티에 이동한다.
4. 카지노에 들어가 환전 ATM으로 현금을 칩으로 바꾼다.
5. 홀짝 테이블에 앉아 베팅을 예약한다.
6. 라운드를 정산 또는 환불로 반드시 닫는다.
7. ledger와 HUD 잔고가 일치하는지 확인한다.
8. 폰/DIS에서 현재 장소, 시간, 채널, 경제 이벤트 흔적을 확인한다.

이 루프가 안정되기 전에는 주식, 유저 간 송금, 노가다, 경마 정산, 관리자 지급, 차량 경제를 붙이지 않는다.

## 데이터 주인

각 데이터는 한 곳만 원본이어야 한다.

| 영역 | 원본 | 원칙 |
| --- | --- | --- |
| 맵/도시 | `map-registry`, map data, editor draft | 맵별 draft key 분리 |
| 플레이어 위치/장소 | `playerState` | `mapId`, `sceneId`, `mode`, `chatChannelId` 함께 변경 |
| 돈/칩/아이템 | economy ledger | UI가 직접 확정하지 않음 |
| 라운드 | round state | 예약 후 정산/환불로 반드시 닫힘 |
| 시간 | `WorldClock`, 향후 서버 시간 | 경제 결과에는 클라이언트 시간 권위 금지 |
| 폰/인터넷 | runtime snapshot/read model | 실제 돈을 직접 바꾸지 않음 |
| 온라인 | future server authority | 클라이언트는 요청, 서버가 effect 확정 |

## 재설계 게이트

다음 기능은 아래 게이트를 통과해야 붙인다.

### 경제 게이트

- ledger entry type이 있는가?
- source 또는 sink가 명확한가?
- 같은 돈 이동이 중복 실행되어도 막히는가?
- 새로고침 후 복구 기준이 있는가?
- 온라인 전환 시 서버가 같은 결정을 내릴 수 있는가?

### 시간 게이트

- 이 기능이 `WorldClock`을 단순 표시로만 쓰는가, 경제 결과에 쓰는가?
- 경제 결과라면 서버 시간 기준으로 바꿀 수 있는가?
- tick 주기가 1분/초 기준에서 너무 빠르거나 느리지 않은가?

### 온라인 게이트

- `mapId`, `roomId`, `channelId`가 명확한가?
- city/venue/table/spectator/admin 상태가 섞이지 않는가?
- 서버가 거절했을 때 UI가 되돌릴 수 있는가?

### UI 게이트

- 모바일에서도 버튼이 겹치지 않는가?
- PC `E`와 모바일 `ACTION`이 같은 action을 만드는가?
- 폰 앱은 정보 read model인지, 돈을 바꾸는 write surface인지 구분되는가?

## 지금 하지 말 것

- 유저 간 송금 구현
- 주식 매수/매도 구현
- 경마 결과/배당 구현
- 노가다 보상 구현
- 편의점 음식 구매 구현
- 배고픔/에너지 자동 감소 구현
- 관리자 지급/차감 구현

위 기능들은 중요하지만 첫 루프가 닫힌 뒤 붙인다. 특히 송금/주식/관리자 지급은 서버 권위와 감사 로그 전에는 구현하지 않는다.

## 지금 해야 할 것

1. 홀짝 UI가 `bet_reserved -> bet_settled | bet_refunded`로 라운드를 닫게 한다.
2. 같은 `roundId`가 두 번 닫히지 않게 round state를 둔다.
3. clean/stale localStorage 브라우저 워크플로우를 점검한다.
4. 첫 루프 기준으로 HUD, 폰, 채팅, 환전, 홀짝이 모바일에서 겹치지 않는지 확인한다.
5. 그 다음에 편의점 음식 구매의 원자적 ledger type을 설계한다.

## 설계 판단 규칙

새 아이디어가 나오면 아래 질문으로 판단한다.

```text
이 기능이 첫 완성 루프를 닫는가?
아니면 루프가 닫히기 전에 경제/온라인/저장 리스크를 키우는가?
```

첫 질문에 답하지 못하면 보류한다. 배금도시 v2는 큰 도시를 한 번에 만드는 게임이 아니라, 작은 도시 루프를 안전하게 닫아가며 커지는 게임이다.
