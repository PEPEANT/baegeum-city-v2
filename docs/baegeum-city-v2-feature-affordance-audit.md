# 배금도시 v2 기능 감각 검증

결론: 건물과 앱은 그래픽 오브젝트가 아니라 플레이어가 기능을 예측하는 약속이어야 한다. 그래서 앞으로 새 건물이나 앱을 추가할 때는 외형보다 먼저 `간판`, `입구`, `기능`, `저장/결과`가 검증에 잡혀야 한다.

## 목적

현재 도시는 건물 수가 늘었지만, 생활 건물 상당수가 `building_shell` 상태라 입장, 내부 기능, 경제 효과가 없다. 스마트폰도 DIS는 미리보기이고 나머지 앱은 잠김 상태다. 이 상태를 디자인 감으로 넘기면 “버그인지 미완성인지” 구분이 안 된다.

따라서 `src/systems/city-feature-audit.js`가 런타임 맵과 폰 앱 카탈로그를 읽어 현재 완성도를 숫자로 보여준다.

## 현재 검증 항목

건물:

- 전체 건물 수
- 입장 가능한 건물 수
- 시각 전용 건물 수
- 간판 없는 건물 수
- 입구 없는 건물 수
- 기능 없는 건물 수

앱:

- 앱 수
- 열 수 있는 앱 수
- 실제 작동 앱 수
- 미리보기 앱 수
- 잠긴 앱 수
- 결과 버튼이 있는 앱 수
- 저장 연동 앱 수

## 현재 판정

- `dice-city`의 복사 카지노는 입장 가능한 `venue_anchor`로 유지한다.
- `baegeum-city` 생활/인프라 건물은 대부분 의도적으로 `building_shell`이므로 검증 패널에서 기능 없는 건물로 잡히는 것이 정상이다.
- DIS는 읽기 미리보기만 가능하고 글쓰기/홈 버튼은 비활성으로 표시한다.
- 은행, 뉴스, 앱스토어, 전화, 갤러리, 증권은 아직 잠김 앱이다.

## 다음 기준

건물 하나를 A급 기능 건물로 올리려면 다음 5개가 있어야 한다.

- 밖에서 정체가 보이는 간판
- 들어갈 수 있음을 알 수 있는 입구
- PC `E` / 모바일 `ACTION` 후보
- 버튼 실행 결과
- DP, 시간, 스탯, 인벤토리, 저장 중 하나 이상에 반영되는 효과

앱 하나를 작동 앱으로 올리려면 다음 5개가 있어야 한다.

- 앱 홈 화면
- 현재 상태 표시
- 실제 누를 버튼
- 버튼 결과 로그
- 다시 열어도 남는 저장 상태

## 다음 구현 순서

1. 편의점 또는 맥버거 같은 생활 건물 1개를 A급으로 만든다.
2. 해당 건물의 입구/상호작용/내부 UI/ledger 효과를 붙인다.
3. 연결 앱을 하나 정해서 버튼 결과와 저장 상태를 붙인다.
4. 검증 패널의 `functionless`와 `workingButtonApps` 수치가 실제로 개선되는지 확인한다.

## 금지

- `building_shell`에 임시로 돈/문/채널을 몰래 붙이지 않는다.
- 앱 버튼이 작동하지 않는데 작동하는 것처럼 보이게 두지 않는다.
- 새 디자인을 예쁘게 칠하기 전에 검증 수치가 무엇을 뜻하는지 먼저 고정한다.
